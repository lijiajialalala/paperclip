## 一句话判断

研究团队已经开始从 6 lane 常驻往新内核收口。2026-04-19 起，live 托管指令已改成 `Research Lead + Challenger` 默认、`Editor / Evidence Auditor / Research Worker / 研究审查官` 按需开启。  
当前最关键的两个 issue 都在 `blocked`，而且没有真实 `executionRunId`，所以这条线的核心问题不是慢，而是没有真正接着跑。

## 这支团队现在实际怎么跑

- 当前 agent roster 仍然是 `研究负责人 + 主研究员 + 交叉研究员 + 证据审校员 + 报告架构师 + 研究审查官`
- 但 2026-04-19 起，live 托管指令已经把这套角色语义改成 `Research Lead + Challenger` 默认，其余按条件启用
- 真正的派工路径还没经过下一轮真实研究任务验证，所以目前不能直接宣称它已经完全切换成功
- 共享状态还没有正式收口成 blackboard 契约
- Clarification Gate 已写进 live 指令，但还没有在新任务里跑过一轮

## 这支团队当前最明显的偏差

- lane 太多，默认就很重
- 单一对外 owner 已写进 live 指令，但还没经过真实任务验证
- `研究审查官` 这个 agent 仍在 roster 里，仍需验证后续任务不会再把它当默认常驻 lane
- 共享材料还没有变成统一黑板，容易重复劳动和上下文漂移

## 新线程接手时最该先确认什么

接手研究线时，先确认这 4 件事：

1. 当前要接的是哪一个研究项目，而不是直接看某个 lane 的最后一句话。
2. `blocked` 的 issue 是否已经恢复真实 run，不要只看 issue 状态。
3. 这轮研究到底是继续旧稿，还是新一轮独立任务。
4. 当前是不是仍然需要 6 lane 全开，还是应该先收回到轻量编排。
