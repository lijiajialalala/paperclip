## 这份文档是干什么的

这不是研发团队“当前怎么跑”的记录。  
这是一份目标内核文档，回答的是：

研发团队接下来应该稳定成什么样，才不会继续卡在负责人单点过重、`in_progress` 漂移和验收边界不清上。

如果你要看线上真实情况，先看 [研发团队当前现状](./研发团队当前现状)。  
如果你要抽象通用字段，再看 [团队配置模板](./团队配置模板)。

## 一句话目标

研发团队的默认编排应该收敛为：

`Tech Lead + Implementer + Acceptance Gate (+ PM / Specialist)`

而不是“所有事都先堆给一个负责人，再靠状态猜谁在做”。

## 目标结构

### 默认角色

| 角色 | 是否默认开启 | 职责 |
| --- | --- | --- |
| `Tech Lead` | 是 | 唯一对外 owner，负责需求澄清、设计、拆解、收口 |
| `Implementer` | 是 | 负责实现和测试，可按前端 / 后端 / 数据 / 客户端拆成多个 worker |
| `Acceptance Gate` | 是 | 独立给出通过 / 不通过，不混入实现 |
| `PM` | 否 | 只在需求口径、验收标准复杂时启用 |
| `Specialist` | 否 | 安全、迁移、发布、性能等专项角色，只在对应风险出现时启用 |

### 当前角色如何收敛

| 当前角色 | 目标去向 |
| --- | --- |
| `技术负责人` | 继续保留，收敛成真正的 `Tech Lead` |
| `产品经理` | 改成条件角色，不再默认和所有研发任务绑定 |
| `前端工程师` / `后端工程师` | 归入 `Implementer` 集合，按任务域启用 |
| `验收审查官` | 保留，但语义统一成 `Acceptance Gate`，不再绑死 QA 命名 |

## 五条硬规则

### 1. 单一对外 owner

- 一个父任务只允许一个对外 owner
- 默认只能是 `Tech Lead`
- 实施角色和验收角色都不能直接向用户要决策
- 缺事实或决策时，只能上浮给 `Tech Lead`

### 2. 计划先于实现

- 手工创建、由负责人治理的实现任务先 `propose-plan`
- `originKind = routine_execution` 的预设 routine lane 默认不走 `work_plan`；如果某支团队想保留 plan gate，必须在团队合同里显式写明
- 对需要 `work_plan` 的任务，批准前不允许编码
- `run 成功` 不等于 `业务通过`
- 验收通过前，父任务不能当成 done

### 3. 验收和实现必须分层

- `Implementer` 不给自己验收
- `Acceptance Gate` 只给 verdict，不接实现
- 验收产物必须独立存在，不准只靠零散评论

### 4. 默认串行，接口稳定后才并行

- 小题默认 `Tech Lead -> Implementer -> Acceptance`
- 只有当边界、接口、验收标准已经稳定，才拆多 lane 并行
- 大任务拆解优先按领域边界，不要只按前后端名义硬拆

### 5. 父子链恢复必须有 owner

- 子任务 done / blocked 后，必须明确唤醒谁来接
- 默认回到 `Tech Lead`
- 不允许父任务长期停在 `in_progress`，但底层没有 active run

## 默认执行链

1. `Tech Lead` 澄清需求和边界
2. 形成 `plan` / `design` / `test-plan`
3. 按需要拆给一个或多个 `Implementer`
4. `Implementer` 完成实现和自测
5. `Acceptance Gate` 给出正式 verdict
6. `Tech Lead` 统一收口并关闭父任务

## 建议共享工作区

默认至少保留这些 keyed documents 或 work products：

| 键 | 用途 |
| --- | --- |
| `plan` | 执行计划 |
| `design` | 技术设计和边界 |
| `test-plan` | 验证和回归范围 |
| `implementation-notes` | 实施 handoff |
| `acceptance-report` | 验收 verdict |
| `release-notes` | 对外或对内变更摘要 |

## 默认角色预算

| 任务规模 | 默认角色 |
| --- | --- |
| 小题 | `Tech Lead + Implementer + Acceptance Gate` |
| 中题 | `Tech Lead + 2 Implementers + Acceptance Gate` |
| 高风险题 | 上述基础上按需加 `PM` / `Specialist` |

不允许因为任务大就先把所有角色全开。

## 研发团队最怕什么

- `in_progress` 看起来正常，其实没有 active run
- 负责人一个人串行卡住全队
- 验收还是挂在旧 QA 命名下，边界越来越模糊
- 子任务做完后没人恢复父链

## 值得后续平台化的部分

| 能力 | 为什么值得平台化 |
| --- | --- |
| execution truth repair | 解决 `checkoutRunId` 和 `executionRunId` 长期漂移 |
| acceptance contract | 让验收真正成为通用 gate，而不是角色习惯 |
| resume-chain ownership | 让父子任务恢复不再靠负责人手工补 |
| lane dispatch policy | 大任务并行何时开启、如何恢复，最好有统一 contract |
