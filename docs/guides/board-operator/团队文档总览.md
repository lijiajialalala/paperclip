## 新线程先看这个

如果你是第一次接手团队相关工作，先不要乱翻文档。  
按下面顺序看：

1. 先看这页。
2. 再看 [团队当前现状](./团队当前现状)。
3. 然后进入对应团队自己的现状文档。
4. 如果你要改研究团队，先看 [研究团队目标内核](./研究团队目标内核)。
5. 如果你要改研发或质量团队，也先看各自的目标内核。
6. 如果你要知道下一步先改什么，再看 [团队收敛路线图](./团队收敛路线图)。
7. 只有在你要抽象成通用模板，或新建别的团队时，才看 [团队配置模板](./团队配置模板)。

| 文档 | 用途 | 什么时候看 | 谁维护 |
| --- | --- | --- | --- |

| [研究团队目标内核](./研究团队目标内核) | 研究线接下来应该稳定成什么样 | 想先把研究团队目标结构定清时 | 改研究团队设计的人 |
| [研发团队当前现状](./研发团队当前现状) | 研发线的真实运行状态和偏差 | 想接手平台开发/修复/验收时 | 改研发团队真实配置的人 |
| [研发团队目标内核](./研发团队目标内核) | 研发线接下来应该稳定成什么样 | 想先把研发团队目标结构定清时 | 改研发团队设计的人 |
| [平台质量团队当前现状](./平台质量团队当前现状) | 质量线的真实运行状态和偏差 | 想接手质量批次/周报/审计时 | 改质量团队真实配置的人 |
| [平台质量团队目标内核](./平台质量团队目标内核) | 质量线接下来应该稳定成什么样 | 想先把质量团队目标结构定清时 | 改质量团队设计的人 |
| [团队收敛路线图](./团队收敛路线图) | 把现状收敛到目标的差距、优先级和下一步动作 | 想知道“现在先改什么”时 | 改团队推进顺序的人 |
| [团队配置模板](./团队配置模板) | 作为新团队或重构团队的参考模板 | 想设计团队时 | 改团队架构的人 |

一句话区分：

- 想知道“现在是什么”先看“团队当前现状”，再进具体团队档案
- 想把研究团队先设计清楚，就先看“研究团队目标内核”
- 想把研发或质量团队设计清楚，也先看各自目标内核
- 想知道“现在先改什么”，就看“团队收敛路线图”
- 想知道“应该怎么抽象成通用表”，再看“配置模板”

## 当前先找谁

| 事情类型 | 先找谁 |
| --- | --- |
| 调研、研究、成稿、研究收口 | `研究负责人` |
| 技术设计、开发推进、验收收口 | `技术负责人` |
| 平台质量批次、质量周报、质量台账 | `质量负责人` |
| 跨团队拍板 | `CEO` |

## 新线程最常犯的错

不要再犯下面这些错：

- 把“模板”当“现状”
- 只看公司级总览，不进具体团队文档
- 看到某个 agent 名字就以为它真的在正常运行
- 只看 issue 状态，不看有没有真实 `executionRunId`
- 只看 routine 存在，不看批次有没有真正被接起来

## 什么时候必须更新文档

出现下面任何一种变化，都要改文档：

- 团队现状入口变了
- 团队负责人变了
- 团队角色变了
- heartbeat 或 routine 配置变了
- 团队工作模式变了
- 团队从参考模式落成了真实固定模式

如果不更新，新线程接手时还是会迷路。
