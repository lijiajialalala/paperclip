## 新线程先看这个

如果你是第一次接手团队相关工作，先不要乱翻文档。  
按下面顺序看：

1. 先看这页。
2. 再看 [团队当前现状](./团队当前现状)。
3. 只有在你要改团队架构时，才看 [团队配置模板](./团队配置模板)。

## 三份文档分别干什么

| 文档 | 用途 | 什么时候看 | 谁维护 |
| --- | --- | --- | --- |
| [团队文档总览](./团队文档总览) | 告诉新线程先看哪份、先找谁 | 第一次接手团队工作时 | 改团队文档结构的人 |
| [团队当前现状](./团队当前现状) | 记录线上真实配置和当前偏差 | 想知道现在到底怎么跑时 | 改真实团队配置的人 |
| [团队配置模板](./团队配置模板) | 作为新团队或重构团队的参考模板 | 想设计团队时 | 改团队架构的人 |

一句话区分：

- 想知道“现在是什么”就看“当前现状”
- 想知道“应该怎么设计”就看“配置模板”

## 当前先找谁

| 事情类型 | 先找谁 |
| --- | --- |
| 调研、研究、成稿、研究收口 | `研究负责人` |
| 技术设计、开发推进、验收收口 | `技术负责人` |
| 平台质量批次、质量周报、质量台账 | `质量负责人` |
| 跨团队拍板 | `CEO` |

## 新线程最常犯的错

不要再犯下面这些错：

- 把“模板”当“现状”
- 看到某个 agent 名字就以为它真的在正常运行
- 只看 issue 状态，不看有没有真实 `executionRunId`
- 只看 routine 存在，不看批次有没有真正被接起来

## 什么时候必须更新文档

出现下面任何一种变化，都要改文档：

- 团队负责人变了
- 团队角色变了
- heartbeat 或 routine 配置变了
- 团队工作模式变了
- 团队从参考模式落成了真实固定模式

如果不更新，新线程接手时还是会迷路。
