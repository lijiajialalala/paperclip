## 这份文档是干什么的

这份文档不记录当前现状，也不重新定义目标内核。  
它只回答一个问题：

从“现在怎么跑”到“应该怎么跑”，三支团队下一步到底要改什么。

如果你要看真实现状，先看各团队“当前现状”。  
如果你要看目标规则，先看各团队“目标内核”。  
如果你要知道“先改哪几件事”，就看这份路线图。

## 读法

建议顺序：

1. 看 [团队当前现状](./团队当前现状)
2. 看对应团队的目标内核
3. 回到这份路线图，看差距和下一步动作

## 优先级标记

| 标记 | 含义 |
| --- | --- |
| `P0` | 不先改，团队几乎无法稳定运行 |
| `P1` | 不先改，会持续漂移或高频返工 |
| `P2` | 应该改，但不是当前最先的阻塞项 |

### 研究团队下一步就该做的事

1. 把研究团队默认出勤模式改成 `Lead + Challenger`，中题再加 `Editor`。
2. 用一轮真实研究任务验证 `研究负责人` 是否真的按新 live instructions 先做 Clarification Gate，再决定 lane 规模。
3. 验证 `主研究员`、`报告架构师`、`证据审校员`、`研究审查官` 是否只在明确条件下才被启用。
4. 如果验证通过，再决定是否把黑板键名和角色启用规则继续下沉成平台约束。

### 平台质量团队下一步就该做的事

1. 先把 `平台质量深度审计批次` 从“偶发跑通”推进到“稳定重复跑通”，直接用最新 `CMPA-179` 验证“生成父 issue -> 负责人接单 -> lane 展开 -> 批次收口”。
2. 在质量负责人指令里写死：routine 任务默认不走通用 intake。
3. 固定批次产物键名和日报 / 周报 / 台账同步要求。
4. 当深度审计批次跑稳后，再考虑把 dispatch 从 `event_driven` 推到 `fixed_parallel_lanes`。

## 三条线的共同原则

虽然三支团队的目标不同，但共同的收敛原则其实是一套：

1. 先把目标内核写清，再抽模板。
2. 单一对外 owner 必须明确。
3. 现状不能只看 issue 状态，必须看执行真相。
4. 默认角色要少，条件角色按风险或负载开启。
5. 先用 SOP 跑稳，再决定哪些能力值得平台化。

## 现在最合理的推进顺序

1. 先把研究团队按目标内核跑通一轮真实任务。
2. 同时修研发团队最硬的执行真相问题。
3. 再把平台质量团队的 routine -> batch 接单链跑通。
4. 等三支团队都至少跑通一轮后，再决定哪些规则要下沉成平台能力。
