## 这份文档是干什么的

这不是平台质量团队“当前怎么跑”的记录。  
这是一份目标内核文档，回答的是：

平台质量团队接下来应该稳定成什么样，才不会继续停在“routine 已创建批次，但没人真正接单开工”的半截状态。

如果你要看线上真实情况，先看 [平台质量团队当前现状](./平台质量团队当前现状)。  
如果你要抽象通用字段，再看 [团队配置模板](./团队配置模板)。

## 一句话目标

平台质量团队的默认编排应该收敛为：

`Routine -> Batch Parent Issue -> Quality Lead -> Fixed Lanes -> Batch Verdict`

而不是每次 routine 触发后又回到通用 intake。

## 目标结构

### 默认角色

| 角色 | 是否默认开启 | 职责 |
| --- | --- | --- |
| `Quality Lead` | 是 | 批次 owner，负责范围确认、lane 收口、日报 / 周报输出 |
| `System Verifier` | 是 | 做系统验证和复现 |
| `Root Cause Analyst` | 是 | 做根因分析和影响面判断 |
| `Experience Auditor` | 是 | 做流程 / 体验 / UI 路径审计 |
| `Engineering Auditor` | 否 | 只在需要工程质量结论时启用 |
| `Security / Performance Lane` | 否 | 只在专项批次或高风险场景启用 |

## 五条硬规则

### 1. routine 是正式入口，不是提醒器

- 例行质量任务默认不需要再次向用户澄清固定要求
- routine 触发后应直接进入批次执行
- 不允许每次都回退到通用 intake

### 2. 每次 run 必须生成独立顶层批次父 issue

- 默认 `runIssueMode = top_level_run_issue`
- 不再挂在长期父 issue 下面反复覆盖
- 一个批次一个 verdict，一个批次一套产物

### 3. 稳定批次优先固定 lane

- 日常深度审计这种高重复批次，默认应收敛到 `fixed_parallel_lanes`
- 只有范围明显异常时，才退回人工干预
- lane 的任务边界要固定，不要每次重新解释

### 4. 质量发现必须沉淀

- 发现不能只留在评论里
- 必须同步进台账、风险清单，必要时生成后续缺陷 issue
- 一个批次结束后，必须留下日报或周报，不允许“做了但没成文”

### 5. 质量团队不写业务代码

- 质量线负责发现、验证、归因、上浮和收口
- 不直接替研发修业务代码
- 证据不足只能标观察项，不能强行下结论

## 默认执行链

1. routine 触发
2. 系统自动生成批次父 issue
3. `Quality Lead` 接单并确认覆盖范围
4. 按固定 lanes 并行展开
5. 各 lane 回写 findings
6. `Quality Lead` 统一给出 batch verdict，并更新日报 / 周报 / 台账

## 建议共享工作区

默认至少保留这些 keyed documents 或 work products：

| 键 | 用途 |
| --- | --- |
| `audit-brief` | 本批次覆盖范围和边界 |
| `findings-ledger` | 本批次发现汇总 |
| `batch-report` | 批次主报告 |
| `daily-summary` | 日报 |
| `weekly-summary` | 周报 |
| `risk-sync` | 与 open risks / defect register 的同步记录 |

## 默认批次规则

### 深度审计批次

- 默认开 `Quality Lead + System Verifier + Root Cause Analyst + Experience Auditor`
- 如果要给正式工程质量结论，再加 `Engineering Auditor`

### 周报汇总

- 默认只由 `Quality Lead` 直接收口
- 不再为周报单独开完整 lanes

## 质量团队最怕什么

- routine 活着，但没人真正接批次
- 批次 issue 生成了，却没有 active run
- findings 全散在评论里，没有台账和正式报告
- 一次 routine 结果反复覆盖上次批次

## 值得后续平台化的部分

| 能力 | 为什么值得平台化 |
| --- | --- |
| routine direct-execute contract | 让预设 routine 不再回到通用 intake |
| fixed lane dispatch | 让稳定批次自动展开，而不是每次人工重派 |
| batch artifact manifest | 让日报、周报、台账同步成为正式 contract |
| finding-to-issue handoff | 让已确认问题能规范生成后续研发缺陷 |
