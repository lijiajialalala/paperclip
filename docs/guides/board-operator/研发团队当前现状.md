## 一句话判断

研发团队现在依然是 `技术负责人` 单点编排的 hub-and-spoke 模式。  
问题不在于“完全没人干活”，而在于 issue 状态和真实 run 之间还存在漂移，所以不能只看 `in_progress` 就以为执行链是健康的。

| 验收规则 | 仍主要由 `验收审查官` 给出 pass / fail；acceptance gate 还没有完全抽成通用机制 |

| 成本策略 | 核心角色都在用 `codex_local + gpt-5.4 + xhigh + timeoutSec=0`，也是默认高配 |
| 完成标准 | 现状上是“修复 + 验证 + 验收通过”；但 `in_progress` 目前不能等价视为有 live run |

## 先找谁

- 负责人：`技术负责人`
- 上级：`CEO`
- 新线程接手研发问题时，先找 `技术负责人`，先确认真实执行归属，再看子任务状态

## 这支团队现在实际怎么跑

- 核心还是负责人驱动
- `产品经理`、`前端工程师`、`后端工程师`、`验收审查官` 都围绕 `技术负责人` 转
- 当前还保留独立的验收角色，没有完全抽成通用 acceptance gate
- 大任务的并行拆解能力有，但仍然高度依赖负责人发起和收口

## 这支团队当前最明显的偏差

- 负责人单点过重
- acceptance 还没有完全从 QA 专属命名和路径里抽出来
- 大型任务的专门并行配置还没沉淀成固定模式
- 子任务完成后，父任务恢复仍然太依赖负责人手工介入

## 新线程接手时最该先确认什么

接手研发线时，先确认这 4 件事：

1. 当前 `in_progress` 的 issue 是否真的绑着一个 live run。
2. 这个任务是继续既有父子链，还是已经需要重新起恢复链。
3. 这轮任务到底该走负责人串行编排，还是已经值得切成独立并行 lane。
4. 验收是应该作为通用 gate 收口，还是还在走旧的 QA 专属路径。
