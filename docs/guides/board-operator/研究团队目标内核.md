## 这份文档是干什么的

这不是研究团队“当前怎么跑”的记录。  
这是一份目标内核文档，回答的是：

研究团队接下来应该稳定成什么样，才不会继续掉回 6 lane 常驻、重复研究、乱问用户、产物对外过载的旧模式。

如果你要看线上真实情况，先看 [研究团队当前现状](./研究团队当前现状)。  
如果你要给别的团队复制一套结构，再看 [团队配置模板](./团队配置模板)。

## 一句话目标

研究团队的默认编排应该收敛为：

`Research Lead + Challenger`

中题及以上默认再加 `Editor`，高风险题再加 `Auditor`。

而不是默认 6 条 lane 常驻。

## 目标结构

### 默认角色

| 角色 | 是否默认开启 | 职责 |
| --- | --- | --- |
| `Research Lead` | 是 | 唯一对外 owner，负责题面解释、证据收口、主结论和最终签字 |
| `Challenger` | 是 | 只做反证、边界、替代解释和降强度建议 |
| `Editor` | 中题及以上默认开启 | 只做前台交付整理、去重、重排、压缩 |
| `Evidence Auditor` | 高风险题开启 | 只在高风险题启用，负责证据等级和引用准确性 |
| `Research Worker` | 高负载或特殊采集时开启 | 只在高负载题或特殊采集场景启用，做采集和整理，不拥有最终判断 |

### 当前 6 角色如何收缩

| 当前角色 | 目标去向 |
| --- | --- |
| `研究负责人` | 继续保留，收敛成真正的 `Research Lead` |
| `交叉研究员` | 继续保留，作为 `Challenger` |
| `报告架构师` | 继续保留，作为 `Editor` |
| `证据审校员` | 保留，但改为高风险题才启用的 `Evidence Auditor` |
| `主研究员` | 不再默认常开；要么并入 `Research Lead`，要么只在高负载题作为 `Research Worker` |
| `研究审查官` | 默认停用，不再作为常驻终审 lane |

## 六条硬规则

### 1. 单一对外 owner

- 一个父任务只允许一个对外发问 owner
- 默认只能是 `Research Lead`
- 其他任何子角色都不能直接向用户提问
- 子角色如果卡住，只能向 `Research Lead` 上浮

### 2. Clarification Gate 必须前置

正式研究前，先判定当前卡的是哪一种缺口：

| 类型 | 含义 | 默认动作 |
| --- | --- | --- |
| `fact_gap` | 缺用户独有事实，公开源补不出来 | 由 `Research Lead` 统一向用户追问 |
| `decision_gap` | 缺用户偏好、排序、取舍 | 由 `Research Lead` 统一向用户要决策 |
| `scope_gap` | 研究对象、边界、比较范围还在飘 | 由 `Research Lead` 先收窄题面，再决定是否对外澄清 |
| `success_gap` | 不知道什么样的交付才算真正答题 | 由 `Research Lead` 先把成功标准写进 `brief` 和 `skeleton` |

Clarification Gate 不通过，不允许直接并行开跑。

### 3. 低风险默认假设要显式

- 高风险题：没有用户回复就不能推进
- 中低风险题：`Research Lead` 可以带假设继续，但必须把假设写进黑板和主文开头
- 不允许把默认假设伪装成已确认事实

### 4. 每条结论必须带证据标签

最终稿和后台材料里的结论，必须能标明它属于哪种证据：

- `官方事实`
- `市场信号`
- `社区信号`
- `推断/建议`

如果做不到，结论强度必须下降。

### 5. 前台和后台必须分层

用户默认看到的是前台交付，不应该被后台审计材料淹没。

| 层级 | 默认产物 |
| --- | --- |
| 前台 | `final-report`、`action-memo` |
| 后台 | `source-matrix`、`challenge-memo`、`audit-memo`、`clarification-log` |

### 6. 前台交付必须过质量 gate

- `final-report` 必须逐项回答 `brief` 的核心问题
- 不允许用岗位百科、背景铺垫、重复结论替代真正的决策输出
- 假设、未决争议、证据不足点必须前置可见
- 如果一篇稿子“材料很多但没有回答题目”，就视为未通过

## Source Matrix 规则

不是所有题都要五类源全开。  
研究团队应该按题型用“最小覆盖集”，而不是形式化凑满。

| 题型 | 最小覆盖集 |
| --- | --- |
| 事实型研究 | 官方源 + 市场源 |
| 口碑 / 体验型研究 | 社区源 + 浏览器采样 |
| 个性化决策型研究 | 用户私有上下文必选，再补官方 / 市场 / 社区所需最小集 |
| 高风险对外报告 | 官方源 + 市场源 + 审校所需补充源 |

默认 5 类源如下：

- 官方 / 原始源
- 市场 / 公开网页
- 社区 / 论坛 / 评论区
- 交互式网页 / 登录态 / 站内搜索
- 用户私有上下文

### 必须升级到社区源或浏览器采样的触发条件

只要出现下面任一情况，就必须把 `社区源` 或 `浏览器采样` 纳入最小覆盖集：

- 题目依赖站内搜索结果
- 题目依赖评论区、论坛楼层、社区回复
- 题目依赖登录态页面或平台原生排序
- 搜索引擎对目标平台的召回明显不可靠

也就是说，不能再默认用 Exa 或普通网页搜索硬顶过去。

## Challenge Pass 规则

Challenge 不是第二条主线研究。  
它只回答三个问题：

1. 三个最强反证是什么
2. 哪个结论需要降强度
3. 哪些条件变化会让主结论翻盘

### 回合预算

- 默认只允许 1 轮 challenge
- 高风险题最多 2 轮
- 不允许无限来回拉扯

### 解决协议

`Challenger` 提出异议后，`Research Lead` 只能三选一：

- 接受并修改
- 驳回并补证据
- 标记为未决争议

做出选择后就闭环，进入下一阶段。

## Editor 边界

`Editor` 只负责前台交付，不负责后台文档扩写，也不负责重新研究。

允许做的事：

- 去重
- 重排结构
- 压缩表达
- 把后台材料翻译成用户能直接使用的前台版本

不允许做的事：

- 偷偷扩题
- 重新搜证据
- 变成第二个主研究员

## 建议黑板结构

在不改平台能力的前提下，先把父 issue 的 keyed documents 当黑板用。

建议最少保留这些键：

| 键 | 用途 |
| --- | --- |
| `brief` | 题面、成功标准、边界 |
| `clarification-log` | 已向用户提出和已确认的事实 / 决策 |
| `source-matrix` | 本题最小信息源覆盖集 |
| `skeleton` | 主结论骨架 |
| `challenge-memo` | Challenger 的反证和 Lead 的处理结果 |
| `final-report` | 前台正式主文 |
| `action-memo` | 前台行动版 |
| `audit-memo` | 高风险题才启用的审校记录 |

原始网页和长评论不要到处传。  
只把“证据摘录 + Source_ID + 结论骨架”写进黑板。

## 默认产物流

### 前台

- `final-report`
- `action-memo`

### 后台

- `brief`
- `clarification-log`
- `source-matrix`
- `skeleton`
- `challenge-memo`
- `audit-memo`

## 预算规则

| 题型 | 默认角色 | Challenge 轮数 | 审校 |
| --- | --- | --- | --- |
| 小题 | `Lead + Challenger` | 1 轮 | 不开 |
| 中题 | `Lead + Challenger + Editor` | 1 轮 | 不开 |
| 高风险题 | `Lead + Challenger + Editor + Auditor` | 最多 2 轮 | 开 |

不允许因为题目模糊就无限加 lane。

## 可以先不改平台的部分

下面这些可以先靠团队内核和 SOP 落地：

- 单一对外 owner
- Clarification Gate 的问题分类
- Challenge 轮数和解决协议
- 前后台产物分层
- Source Matrix 的最小覆盖集
- 证据标签

## 值得后续平台化的部分

下面这些长期仍值得升级成平台能力，否则时间长了会漂：

| 能力 | 为什么值得平台化 |
| --- | --- |
| 黑板 manifest | 现在 document 只是通用文本容器，还不是正式黑板 |
| Clarification 挂起 / 超时恢复 | 现在只能靠 blocked、comment、manual wake 软落地 |
| 结构化 citation schema | 现在仍是写作规范，不是系统硬约束 |
| 浏览器失败降级策略 | 现在可以写 SOP，但还没有统一 failure policy |

## 这份内核怎么反推模板

顺序应该是：

1. 先把目标内核写清
2. 再把稳定字段抽成 [团队配置模板](./团队配置模板)
3. 再用同一套字段去推研发团队和平台质量团队

也就是说，模板不是起点，而是目标内核的抽象结果。
