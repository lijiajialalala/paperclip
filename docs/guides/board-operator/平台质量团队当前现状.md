## 一句话判断

平台质量团队已经不只是“方向对了”，而是至少跑通过一轮完整批次。  
但这条链还没有稳定复现：上一轮 `CMPA-170` 已经展开并收完 4 条默认子 lane，最新一轮 `CMPA-179` 却又停在 `todo`。当前最大的断层不是“完全不会跑”，而是“不能稳定重复接单开工”。

## 这支团队现在实际怎么跑

- 这条线挂在 `平台质量运营` 项目下
- 方向上已经切到“routine 生成独立顶层父 issue”
- `CMPA-170` 这轮已经证明批次父 issue 和 4 条默认子 lane 可以完整收口
- 但 dispatch 目前还是 `event_driven`
- specialist lane 已经存在，但 heartbeat 默认关闭或按需唤醒
- 所以它还不是稳定重复运行的 `fixed_parallel_lanes` 批次系统

## 这支团队当前最明显的偏差

- routine 已存在，而且至少跑通过一轮，但执行链不稳
- dispatch 还是 `event_driven`，还没有切到稳定的 `fixed_parallel_lanes`
- specialist lane 都在，但默认不开工
- 从“最新批次 issue 已创建”到“正式开工”之间还有人为恢复断层

## 新线程接手时最该先确认什么

接手质量线时，先确认这 4 件事：

1. 最新 routine 生成的父 issue 是哪一个。
2. 这个父 issue 是否已经挂上真实 `executionRunId`。
3. 这轮是应该由 `质量负责人` 先收口，再拆 lane，还是 lane 已经被稳定触发。
4. 这次发现的问题是要沉淀进批次报告，还是应该直接生成后续缺陷 issue。
